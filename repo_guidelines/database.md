# Database Migrations

Database migration standards for the project. Covers migration tool structure, authorship, naming, startup lifecycle, rollback posture, environment parity, and seed data.

For the architectural principles that motivate these rules, see [architecture.md](./architecture.md).

---

## Guiding Principle

Schema changes are code. They must be reviewed, versioned, and deployed through the same pipeline as application code. Manual DDL applied outside the migration tool is not acceptable.

---

## Migration Tool

Choose a migration tool appropriate for the project's language and ORM. Examples include Alembic (Python/SQLAlchemy), Flyway or Liquibase (JVM), Knex migrations (Node.js), or ActiveRecord migrations (Ruby). The chosen tool must support:

- versioned, sequential migration files,
- forward (upgrade) and reverse (downgrade) operations,
- integration with the project's ORM or schema definitions,
- command-line execution suitable for CI and container entrypoints.

Document the chosen tool and its invocation commands in the project README or `docs/runbooks/local-development.md`.

---

## Project Structure

Migration files and configuration must live inside the backend service, co-located with the application code they evolve alongside.

```text
backend/
  migrations/
    versions/
      0001_create_users_table.<ext>
      0002_add_projects_table.<ext>
    config.<ext>
```

- Database connection details must be resolved through environment variable interpolation, not hardcoded credentials.
- If the migration tool supports autogeneration from model definitions, configure it to reference the application's current model state.

---

## Migration Authorship

- **One migration per logical schema change.** A single migration should represent one coherent change — adding a table, adding a column, creating an index. Do not bundle unrelated changes.
- **Autogenerated migrations are a starting point, not the final product.** Always review autogenerated output before committing. Verify column types, nullable flags, defaults, index names, and constraint behavior.
- **No silent data destruction.** Migrations that drop columns, drop tables, or alter data must document whether the operation is reversible. If it is not, the migration must include a comment explaining why a downgrade path is not provided.
- **Migration files are immutable after merge.** Once a migration has been applied in any shared environment, it must not be edited. Fixes go in a new migration.

---

## Naming and Ordering

- Use descriptive revision messages that communicate intent: `create_users_table`, `make_email_unique`, `drop_legacy_status_column`.
- Prefer a linear migration history. Avoid branching revisions unless the team has tooling and process to resolve merge conflicts safely.

---

## Startup Lifecycle

Migrations must run as a **distinct, pre-application step** — not silently inside the app server's startup code.

**Preferred patterns:**

- A Docker entrypoint script that runs the migration upgrade command before starting the application process.
- A dedicated init container or compose service that completes migrations before the app service starts.
- A CI/CD pipeline step that runs migrations against the target database before deploying the new application image.

**Rules:**

- The migration step must be **visible to operators** in logs and deployment output.
- The application process must not start serving traffic until migrations have completed successfully.
- Migration failures must **halt startup**, not silently continue with an outdated schema.

---

## Rollback Posture

- **Additive migrations** (new tables, new columns, new indexes) should include a downgrade function when practical.
- **Destructive migrations** (drops, renames, type changes) must document their rollback strategy:
  - a working downgrade function, or
  - an explicit comment stating irreversibility and the forward-fix strategy.
- The default recovery strategy is **forward-fix**: deploy a new migration that corrects the issue rather than rolling back.

---

## Environment Parity

- The same migration chain must run in **development, CI, staging, and production**.
- Do not maintain separate migration paths per environment.
- Environment-specific seed data or fixtures belong in separate scripts or bootstrap logic, not inside migration files.

---

## Production Database Initialization

Applications must **never** receive the database root password. In production, infrastructure-level database objects — the application database, the application user, and privilege grants — must be created by an administrator before the application is deployed.

Each project must maintain a **`db.init.sql`** file at:

```text
infra/
  db.init.sql
```

This file contains the SQL statements an administrator runs **once** during initial infrastructure setup, using root or admin-level credentials that the application itself never possesses.

**`db.init.sql` must include:**

- `CREATE DATABASE` (with `IF NOT EXISTS`)
- `CREATE USER` (with `IF NOT EXISTS`)
- `GRANT` statements scoped to the application database only (principle of least privilege)

**Example:**

```sql
-- db.init.sql — Run by infrastructure admin with root credentials.
-- Application services must NEVER have access to this password.

CREATE DATABASE IF NOT EXISTS myapp;
CREATE USER IF NOT EXISTS 'myapp_user'@'%' IDENTIFIED BY '${DB_APP_PASSWORD}';
GRANT ALL PRIVILEGES ON myapp.* TO 'myapp_user'@'%';
FLUSH PRIVILEGES;
```

**Rules:**

- `db.init.sql` is **not executed by the application, migration tool, or container entrypoint**. It is run manually (or via infrastructure automation such as Terraform/Ansible) by an operator with elevated credentials.
- The file must use **placeholder variables** (e.g., `${DB_APP_PASSWORD}`) for secrets rather than hardcoded values.
- The file must be **idempotent** — safe to run multiple times without error (use `IF NOT EXISTS`).
- `db.init.sql` handles only infrastructure-level concerns (database, user, grants). Schema creation belongs exclusively in migrations.
- In **local development**, Docker Compose handles database initialization automatically (e.g., MySQL/Postgres init scripts mounted into the container), so `db.init.sql` is **not used locally**.

---

## Seed and Bootstrap Data

- Initial admin users and required reference data should be handled by the **application bootstrap lifecycle** (see [auth.md — Admin Seeding](./auth.md#admin-seeding)), not by migration files.
- Migrations may insert data only when it is a **structural requirement** of the schema change (e.g., populating a new enum table that a foreign key depends on).
- Seed scripts must be idempotent.

---

## Reject Immediately

- Manual DDL applied to any environment outside of the migration tool.
- Autogenerated migrations committed without review.
- Migrations that silently drop data without documentation.
- Migration upgrade commands buried inside the application server's startup with no operator visibility.
- Different migration chains or skipped migrations across environments.
- Business logic or data transformations embedded in migration files when they belong in application-level scripts.
